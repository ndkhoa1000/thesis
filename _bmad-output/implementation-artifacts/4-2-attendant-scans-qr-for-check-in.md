# Story 4.2: Attendant Scans QR for Check-In

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As an Attendant,
I want to scan a driver's check-in QR code,
so that I can register the vehicle's entry in under 5 seconds.

## Acceptance Criteria

1. Given I am authenticated with a dedicated Attendant account assigned to a parking lot, when I open the attendant gate flow, then I can access a scanner-first check-in screen scoped to my assigned lot instead of the current placeholder attendant shell.
2. Given I scan a valid driver check-in QR generated by Story 4.1, when the backend validates the token, then the system creates exactly one `ParkingSession` linked to the attendant's lot, selected driver, and selected vehicle.
3. Given the check-in succeeds, when the session is created, then `current_available` is updated in the same transaction and the attendant sees immediate success feedback suitable for gate use.
4. Given the scanned QR is invalid, expired, malformed, or already unusable because the driver already has an active session, when I attempt the scan, then the system rejects the check-in with a clear attendant-facing error and does not create a duplicate session.
5. Given this story only covers app-driver QR entry, when it is implemented, then NFC fallback, walk-in/manual plate capture, and payment/check-out flows remain out of scope for later stories.

## Implementation Clarifications

- This story is the first place where the real `ParkingSession` is created for the driver QR path. Story 4.1 must remain pre-session only.
- Story 4.2 consumes the backend-signed pre-session token emitted by `POST /sessions/driver-check-in-token`; it must not trust raw client-built vehicle or driver fields.
- The current mobile Attendant experience is only a placeholder in `main.dart`. This story should replace that placeholder with a minimal scanner-first attendant workspace, but it must not assume the full Epic 9 shell already exists.
- QR-only scope is intentional here. Do not pull NFC fallback into this story even though the broader product vision mentions it.
- Walk-in/manual check-in and attendant photo-capture-heavy flows belong to Story 4.3. Do not collapse Story 4.2 and 4.3 together.
- Driver checkout QR generation and payment resolution belong to Stories 4.4 through 4.6.

## Dev Notes

### Story Foundation

- Story 4.1 already established the pre-session contract: drivers generate a short-lived backend-signed check-in token, and no `ParkingSession` is created until this story consumes that token.
- Story 3.3 already provisions dedicated `ATTENDANT` credentials tied to a single `Attendant(user_id, parking_lot_id)` profile, which is the lot context this story must use.
- The backend currently has `ParkingSession` and `ParkingLot.current_available`, but no Attendant-facing session creation contract exists yet.
- The mobile app currently routes Attendant logins to a placeholder text screen only; there is no scanner feature or attendant gate UI implemented yet.

### Domain Constraints To Respect

- Attendant accounts remain separate sessions and are not part of the public Driver/LotOwner/Operator workspace-switching flow.
- The attendant must only create sessions for the lot assigned through the active `Attendant.parking_lot_id` profile. The scanned QR does not choose the lot.
- Check-in must reject drivers who already have an active session at the same lot or at another lot, matching the PRD edge-case intent.
- Session creation and lot availability updates must be atomic to satisfy NFR15 and prevent gate-state drift.
- This story covers the app-driver QR path only. NFC fallback, walk-in capture, and incomplete-media remediation should remain outside this story's scope unless absolutely required for the QR path to function.

### Architecture Guardrails

- Backend implementation should stay inside `backend/src/app/api/v1/sessions.py` and related session schemas; do not create a separate temporary attendant route family for this flow.
- Validate the driver check-in token server-side using the same backend signing model introduced in Story 4.1. Do not trust decoded client data without backend verification.
- If the current auth payload does not expose enough attendant lot context for mobile UX, prefer adding a focused backend-read path or enriching the existing safe auth summary rather than hardcoding a lot ID in the client.
- `current_available` must change within the same transaction as `ParkingSession` creation. Avoid split writes that can leave capacity inconsistent after failures.
- Keep the mobile Attendant UI implementation self-contained and injectable from the current `session.isAttendant` branch in `mobile/lib/main.dart`, without assuming Epic 9 workspace-shell infrastructure.

### Library / Framework Requirements

- Flutter currently does not include a QR scanner dependency. Use a proven scanner package suitable for fast gate scanning, such as `mobile_scanner`, only when implementing the story.
- Continue using FastAPI route dependencies and database ownership checks for authorization. Do not bypass the existing auth model with client-side role assumptions.
- Do not add backend image-generation or NFC-specific dependencies for this story.

### File Structure Requirements

- Backend:
  - `backend/src/app/api/v1/sessions.py`
  - `backend/src/app/schemas/session.py`
  - `backend/src/app/models/sessions.py`
  - `backend/src/app/models/parking.py`
  - `backend/tests/test_attendant_check_in.py` (preferred focused suite)
- Mobile:
  - `mobile/lib/main.dart`
  - `mobile/lib/src/features/attendant_check_in/` (new feature folder if needed)
  - `mobile/lib/src/features/auth/data/auth_service.dart`
  - `mobile/test/` focused attendant widget test file

### Testing Requirements

- Backend verification should run through the Docker pytest service using the repo convention for targeted suites.
- Flutter verification should run with local `flutter test`.
- Android/manual gate-device smoke validation remains user-owned unless explicitly requested.

### Implementation Risks To Avoid

- Do not recreate the Story 4.1 driver QR generation logic inside the attendant flow.
- Do not create a session without also updating lot availability atomically.
- Do not leave the Attendant app on a generic placeholder once this story is implemented; Story 4.2 must provide a real gate-facing screen.
- Do not pull manual plate entry, walk-in camera capture, checkout pricing, or payment settlement into this story.
- Do not assume the driver's QR token alone is a persistent session QR. Story 4.2 is the session-creation boundary; later stories handle checkout-state behavior.
