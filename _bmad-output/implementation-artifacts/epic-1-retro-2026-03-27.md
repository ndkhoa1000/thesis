## Part 1: Epic Review

### What Went Well

- Epic 1 established the foundation correctly: one shared public account, additive public capabilities, and separate Attendant/Admin credentials. That decision stayed consistent across Stories 1.1 to 1.5 and prevented later auth rewrites.
- The team reused the existing FastAPI boilerplate incrementally instead of attempting a disruptive rewrite. This matched the architecture and kept the repo runnable while features were added.
- Cross-layer delivery improved through the epic. Early stories built auth bootstrap and session handling; later stories reused that work effectively for vehicle management and capability application flows.
- Story 1.4 became a strong reusable pattern. Dedicated application records, admin review endpoints, session refresh behavior, and focused mobile status screens gave Story 1.5 a proven implementation baseline.
- Verification discipline improved. Docker-based backend tests, local Flutter widget tests, and Android smoke validation became part of the working pattern rather than ad hoc afterthoughts.

### What Did Not Go Well

- Artifact and tracker hygiene lagged behind implementation. Story files, task checklists, and sprint-status entries were not consistently updated when work moved from implementation to review to closure.
- Several real issues were only caught after manual runtime testing, not by the first pass of automated tests. The Operator form validation mismatch and the approved LotOwner-to-Operator routing bug are the clearest examples.
- The mobile shell is still role-first and placeholder-heavy. That was acceptable early in the epic, but it became a structural constraint once additive capabilities started to accumulate.
- Scope boundaries were not always explicit enough. Story 1.5 exposed ambiguity between “backend approval flow exists” and “approval flow is fully usable inside the app,” which forced planning decisions during review instead of before implementation.
- Admin behavior is partially modeled in auth payloads but not fully aligned in route dependencies. That inconsistency did not block Epic 1 implementation entirely, but it will become a blocker for Epic 2 if not addressed first.

### Recurring Patterns Across Stories

- Contract alignment matters: when backend schema constraints and Flutter form validation diverged, runtime failures escaped despite passing tests.
- Review feedback was most valuable when it targeted state transitions and lifecycle integrity, not just happy-path behavior.
- Reusable implementation patterns paid off quickly. Story 1.5 moved faster because Story 1.4 had already clarified how capability applications should work.
- The project is now sensitive to shell architecture. As soon as one public identity can hold Driver, LotOwner, and Operator capabilities, relying on a single `user.role` landing path starts creating UX and routing debt.

### Root Lessons Learned

1. Definition of Done must include artifact synchronization, not just passing code and tests.
2. Applicant-side success is not enough for approval workflows; the team must decide explicitly whether “done” includes app-facing admin operations.
3. Additive-capability systems need state-transition tests early. Role promotion and workspace resolution are first-class behaviors, not secondary UI details.
4. Mobile forms should mirror backend validation rules from the start to avoid false-success UX and HTTP 422 surprises.
5. When introducing new persistence shapes, checked-in migrations and preserved history should be treated as part of the initial implementation, not review cleanup.

## Part 2: Next Epic Preparation

### Next Epic Readiness

- Epic 2 is defined and already contains a follow-up Story 2.2 for the missing app-facing Admin approvals dashboard.
- Epic 2 depends directly on Epic 1 outcomes, especially the LotOwner capability flow, Operator capability flow, and admin approval semantics.
- The most important readiness issue is not raw feature count. It is consistency: admin authorization, app shell behavior, story status tracking, and review/approval expectations must be normalized before Epic 2 expands the system further.

### Critical Discoveries That Affect Epic 2

1. The current Admin approval path exists in backend APIs but not in the mobile app, so Epic 2 should not assume approval workflows are already user-complete.
2. Admin capability detection and admin route protection are still misaligned (`role == ADMIN` vs `is_superuser`). Epic 2 should fix this before building more admin-facing flows.
3. Public workspace routing remains fragile for multi-capability users. Epic 2 can proceed, but more role-based features will compound this debt if the shell stays unchanged.

### Team Agreements For Future Epics

- Do not leave story artifacts behind the codebase. If a story changes state, the artifact changes in the same pass.
- Treat manual device validation as a required signal for mobile stories that touch end-to-end workflows.
- When a story introduces a reusable pattern, name it and reuse it deliberately in the next related story.
- Raise scope ambiguity during story preparation, not during review after implementation exists.

### Preparation Recommendations For Epic 2

- Prioritize Story 2.2 as the first explicit place to close the admin approval loop in-app.
- Resolve the Admin auth mismatch before adding more protected admin views.
- Keep the deferred UX hardening work visible, but do not let shell standardization block the immediate approval workflows.
- Revisit public workspace switching before Epic 3 expands Operator behavior further.

## Summary

Epic 1 succeeded at establishing the shared-account identity model and proving that the backend/mobile stack can deliver full vertical slices. The strongest positive pattern was incremental reuse of architecture-aligned building blocks. The strongest warning sign was process drift: implementation moved faster than artifact maintenance and faster than some runtime-quality checks. The next epics should preserve the technical direction from Epic 1, while tightening closure discipline, approval-flow scoping, and multi-capability shell behavior.
